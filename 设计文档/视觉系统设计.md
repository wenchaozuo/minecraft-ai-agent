## 一、为什么需要视觉系统

### 1.1 当前感知的局限

现有感知层完全基于 mineflayer API 的文本数据，缺少对空间结构、建筑风格、地形特征的理解：

| 场景 | 文本感知（现状） | 视觉感知（目标） |
|------|-----------------|-----------------|
| 玩家建筑 | `oak_planks x N, glass_pane x N` | "一栋带玻璃窗的木屋" |
| 洞穴结构 | `stone, air, stone, air...` | "一个向北延伸的洞穴，有分叉" |
| 地形特征 | `water, grass_block, stone...` | "河流从中间穿过，两岸有树林" |
| 敌情判断 | `zombie at distance 12` | "僵尸在正前方，被树干挡住了" |

### 1.2 视觉能力的定位

视觉系统不是替代文本感知，而是**补充**——文本感知提供精确数据（坐标、类型、数量），视觉提供结构和空间的判断。

```
最终感知 = 精确文本数据 + VLM 视觉理解
```

## 三、渲染管线（JS 层）

### 3.1 数据获取

通过 mineflayer 的 `findBlocks` API 批量采集周围方块数据：

```javascript
// 核心数据采集接口
function collectBlocks(bot, radius = 30, height = 12) {
  const origin = bot.entity.position

  // 使用 findBlocks 替代暴力三重循环，性能高一个数量级
  const blocks = bot.findBlocks({
    matching: block => block.name !== 'air',
    maxDistance: radius,
    count: 5000
  })

  return blocks.map(pos => ({
    name: bot.blockAt(pos).name,
    x: pos.x - Math.floor(origin.x),
    y: pos.y - Math.floor(origin.y),
    z: pos.z - Math.floor(origin.z)
  }))
}
```

**采集策略**：

| 范围 | 方块数 | 触发条件 | 用途 |
|------|--------|---------|------|
| 20×20×8 | ~1,200 | 常规感知 | 环视、快速分析 |
| 40×40×12 | ~5,000 | 探索新区 | 建筑识别、地形建模 |
| 60×60×16 | ~10,000 | 第一次到达新区域 | 大地图建图 |

### 3.2 Three.js 场景构建

在 Node.js 中用 `three` 包构建 3D 场景。不需要浏览器和 prismarine-viewer。

```javascript
const THREE = require('three')
const { createCanvas, loadImage } = require('canvas')

// 场景初始化
const scene = new THREE.Scene()

// 光照
scene.add(new THREE.AmbientLight(0xffffff, 0.6))         // 环境光
scene.add(new THREE.DirectionalLight(0xffffff, 0.8))     // 方向光

// 相机（初始化时设置，每次渲染调整位置）
const camera = new THREE.PerspectiveCamera(75, 1, 0.5, 200)
```

**核心渲染流程**（伪代码）：

```javascript
async function renderBlocks(blocks, textures, cameraConfig) {
  // 1. 清空场景
  scene.children.filter(c => c.isMesh).forEach(c => scene.remove(c))

  // 2. 为每个方块创建 Mesh
  //    相同方块类型用 InstancedMesh 批处理
  const groups = groupByType(blocks)
  for (const [type, positions] of groups) {
    const texture = textures[type]
    const material = new THREE.MeshLambertMaterial({ map: texture })
    const geometry = new THREE.BoxGeometry(1, 1, 1)

    for (const pos of positions) {
      const mesh = new THREE.Mesh(geometry, material)
      mesh.position.set(pos.x, pos.y, pos.z)
      scene.add(mesh)
    }
  }

  // 3. 设置相机
  camera.position.set(cameraConfig.x, cameraConfig.y, cameraConfig.z)
  camera.lookAt(cameraConfig.target)

  // 4. 渲染到 canvas
  renderer.render(scene, camera)

  // 5. 导出 PNG
  return canvas.toBuffer('image/png')
}
```

### 3.3 纹理加载

#### 3.3.1 纹理来源

纹理文件已放置在 `textures/block/` 目录下，共 1,111 张 PNG，来自 Minecraft Java Edition 原版纹理包。

#### 3.3.2 方块名到纹理文件的匹配规则

mineflayer 返回的方块名和文件名 95% 以上直接对应，不需要手工映射表：

```
规则一：直接匹配
  oak_planks  → textures/block/oak_planks.png  ← fs.existsSync 验证

规则二：文件变体
  不存在的文件名试 _top / _side / _front 后缀

规则三：多面纹理特例表（约 30 个）
  grass_block:  { top: grass_block_top, side: grass_block_side, bottom: dirt }
  oak_log:      { top: oak_log_top,     side: oak_log }
  crafting_table: { top: crafting_table_top, side: crafting_table_side, front: crafting_table_front }
  ...

兜底：没找到纹理的方块用 stone 纹理替代
```

#### 3.3.3 纹理加载（Node.js + canvas 包）

```javascript
const { loadImage } = require('canvas')
const textureCache = {}

async function loadBlockTexture(blockName) {
  if (textureCache[blockName]) return textureCache[blockName]

  const config = resolveTexturePath(blockName)  // 按规则找文件
  const img = await loadImage(config.path)
  const canvas = createCanvas(img.width, img.height)
  const ctx = canvas.getContext('2d')
  ctx.drawImage(img, 0, 0)

  const texture = new THREE.CanvasTexture(canvas)
  texture.magFilter = THREE.NearestFilter  // 保持像素风
  textureCache[blockName] = texture
  return texture
}
```

function renderSurroundings(bot, blocks, textures) {
  const directions = [
    { yaw: 0,   label: '前方' },     // bot 当前朝向
    { yaw: 90,  label: '右侧' },
    { yaw: 180, label: '后方' },
    { yaw: 270, label: '左侧' },

  const tiles = directions.map(dir => {
    const camera = createCamera(bot, dir.yaw, 0)

#### 3.4.3 俯视图（45°）

从空中俯瞰场景，适合建筑识别和地形分析：

```javascript
function renderTopDown(bot, blocks, textures) {
  const pos = bot.entity.position
  const camera = {
    position: { x: pos.x + 30, y: pos.y + 50, z: pos.z + 30 },
    target: { x: pos.x, y: pos.y, z: pos.z }
  }
  return renderBlocks(blocks, textures, camera)
}
```

#### 3.4.4 洞穴切片

只渲染 bot.y ± 3 层的方块，其他层跳过。解决洞穴内被遮挡的问题：

```javascript
function renderCaveSlice(blocks, textures, botY) {
  const sliceBlocks = blocks.filter(b => Math.abs(b.y - botY) <= 3)
  // 用俯视角渲染过滤后的方块
  return renderTopDown(sliceBlocks, textures, cameraConfig)
}
```

## 五、视觉分析流水线（Python 层）

### 5.1 VLM 分析流程

Python 端接收图片后，调用 VLM API 进行分析：

```python
"""
ai/vision/analyzer.py
视觉分析器 - 接收截图，调用 VLM 返回结构化描述
"""

class VisionAnalyzer:
    def __init__(self, model="gpt-4o"):
        self.model = model

    def analyze(self, image_path: str, camera_type: str,
                metadata: dict) -> dict:
        """
        分析截图，返回结构化描述

        Args:
            image_path: 截图文件路径
            camera_type: 视角类型
            metadata: 位置、朝向等信息

        Returns:
            {
                "structures": [...],    # 识别到的建筑
                "terrain": "...",       # 地形描述
                "threats": [...],       # 可见威胁
                "resources": [...],     # 可见资源
                "request_view": None    # 如需其他视角，告诉 JS 层
            }
        """
        prompt = self._build_prompt(camera_type, metadata)
        result = self._call_vlm(image_path, prompt)
        return self._parse_result(result)
```

### 5.3 合并到感知数据

视觉分析的输出不是替代文本感知，而是作为**补充维度**合并：

```python
def merge_perception(text_perception: dict, vision_result: dict) -> dict:
    """
    合并文本感知 + 视觉分析结果

    text_perception 来自 existing getFullPerception():
    { environment, threat, resources, progress, worldMap }

    vision_result 来自 VisionAnalyzer:
    { structures, terrain, threats, resources, notes }
    """
    return {
        # 保留原有文本感知
        "environment": text_perception["environment"],
        "threat": text_perception["threat"],
        "resources": text_perception["resources"],
        "progress": text_perception["progress"],

        # 新增视觉补充
        "vision": {
            "structures": vision_result.get("structures", []),
            "terrain": vision_result.get("terrain", ""),
            "visible_threats": vision_result.get("threats", []),
            "timestamp": time.time()
        },

        # LLM 摘要中加入视觉信息
        "summary": generate_summary_with_vision(
            text_perception, vision_result
        )
    }
```

## 六、渲染性能

### 6.1 渲染频率

| 触发条件 | 频率 | 说明 |
|---------|------|------|
| 常规决策轮询 | 3 秒/次 | 第一人称或环视，低开销 |
| 建筑/区域分析 | 手动触发 | 俯视图，LLM 请求时渲染 |
| 洞穴探索 | 每移动 5 格 | 洞穴切片，随位置更新 |
| 首次到达新区 | 一次 | 大范围渲染，建图用 |

### 6.2 性能预算

| 方块数量 | 纹理加载（首次） | 渲染耗时 | 说明 |
|---------|----------------|---------|------|
| 1,200 | ~50ms | ~30ms | 常规范围，纹理已缓存 |
| 5,000 | ~200ms | ~100ms | 大范围，首次渲染 |
| 10,000 | ~300ms | ~200ms | 极限范围，超过此值应考虑 LOD |

纹理加载是主要耗时点，但只发生在**首次遇到该方块类型时**。纹理缓存后，后续渲染仅需 ~30-100ms。

### 6.3 优化策略

1. **纹理缓存** — `loadBlockTexture` 的结果全局缓存，每个方块类型只加载一次
2. **InstancedMesh** — 同类方块合并 draw call，万级方块也能流畅渲染
3. **范围自适应** — 性能不足时自动缩小渲染半径
4. **离屏渲染** — 渲染在后台线程进行，不阻塞决策循环

## 九、与现有系统的集成

### 9.1 触发机制

视觉分析不持续运行，而是**按需触发**：

```python
# 决策循环中判断是否需要视觉数据
def should_request_vision(perception, current_goal) -> bool:
    reasons = []

    # 1. 首次到达新区域
    if is_new_area(perception):
        reasons.append("new_area")

    # 2. 当前目标涉及结构识别
    if current_goal in ["find_village", "find_stronghold", "build_base"]:
        reasons.append("goal_requires_vision")

    # 3. 检测到可疑环境（太多遮挡、不确定地形）
    if perception["environment"]["isUnderground"]:
        reasons.append("underground")

    # 4. 决策 LLM 主动请求
    if perception.get("vision_requested"):
        reasons.append("llm_requested")

    return reasons
```

### 9.2 感知融合

```
文本感知（高频，精确）
  ├─ health/food/inventory  →  决策阈值判断
  ├─ entity list             →  攻击/躲避决策
  └─ position/维度           →  导航依据

视觉分析（低频，辅助）
  ├─ structure 识别          →  目标/导航点
  ├─ 地形分析               →  路线规划
  └─ 建筑特征               →  回基地、找资源

最终：精确数据 + 空间理解 = 更好的决策
```
